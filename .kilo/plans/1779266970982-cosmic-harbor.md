# GUI Testing Exploration Plan

## Goal
Explore practical GUI testing options for OpenMUX and decide on a small, maintainable first step that fits the current macOS-only, AppKit-first, terminal-first architecture.

## Current Context
- OpenMUX is a Swift Package Manager project with AppKit shell code in `OmuxAppShell` and the terminal runtime isolated behind `OmuxTerminalBridge`.
- Existing tests are mostly Swift unit/integration tests under `Tests/`, including substantial AppKit/controller coverage in `OmuxAppShellTests`.
- There is no OpenMUX UI test target today.
- The repo already has launch-level smoke tests:
  - `Scripts/smoke-openmux-app.sh`
  - `Scripts/smoke-packaged-release-app.sh`
- Vendored Ghostty includes `XCTest` UI tests under `Vendor/ghostty/macos/GhosttyUITests/`, which provide useful patterns but should not become a dependency or copied wholesale.
- The packaged app resource-loading guardrail is important: GUI testing should not only target local `swift run`, but also verify release-style `.app` behavior where practical.

## Options

### Option 1: Stay With AppKit Unit/Integration Tests
Continue testing UI behavior by instantiating controllers/views directly in `OmuxAppShellTests`.

Pros:
- Fast and stable compared with full app UI automation.
- Works naturally with SPM and current test layout.
- Good for menu wiring, view state, accessibility labels, layout/persistence logic, command palette model behavior, and controller mutations.
- Avoids flakiness from real windows, focus, Mission Control, animations, and terminal runtime timing.

Cons:
- Does not prove real app launch, first responder behavior, keyboard delivery, focus, menu routing, or Accessibility tree usability.
- Cannot catch some regressions users experience only through the running app.

Best use:
- Keep this as the default layer for most shell behavior.

### Option 2: Add Native XCTest UI Tests
Add a dedicated UI test target that launches `OpenMUXApp` and drives it through `XCUIApplication`.

Pros:
- Native Apple tooling; no extra runtime dependency.
- Can validate app launch, menu items, command palette open/close, workspace/sidebar buttons, keyboard shortcuts, focus movement, modal/palette behavior, and Accessibility labels.
- Ghostty already demonstrates viable patterns using temporary config files, launch environments, `-ApplePersistenceIgnoreState YES`, and `XCUIApplication` interactions.

Cons:
- Swift Package Manager alone does not model Xcode UI test bundles well; likely needs either an `.xcodeproj`, generated project workflow, or a separate test harness package/app target strategy.
- Full UI tests are slower and more flaky than AppKit tests.
- CI may require a logged-in macOS GUI session and Accessibility/input permissions depending on how tests are run.

Best use:
- A small manually-invoked or CI-gated smoke UI suite covering only critical user flows.

### Option 3: Expand Shell Smoke Scripts With Accessibility Probes
Extend existing smoke scripts to launch the app and query state with macOS tools such as `osascript`, `System Events`, or a small purpose-built Swift CLI using Accessibility APIs.

Pros:
- Fits the existing `make smoke` / packaged smoke workflow.
- Can cover packaged `.app` resource-loading and startup regressions.
- Does not require an Xcode UI test target.
- Useful for coarse checks: app launches, window appears, menu exists, app remains responsive, maybe command palette can be invoked.

Cons:
- AppleScript/System Events can be brittle, permission-sensitive, and weakly typed.
- Harder to express rich assertions than XCTest.
- Accessibility permissions can make CI difficult.

Best use:
- Release smoke tests and coarse packaged-app checks, not detailed interaction tests.

### Option 4: Snapshot/Screenshot Testing
Capture screenshots of key shell states and compare against stored baselines.

Pros:
- Useful for visual regressions in sidebar, pane chrome, command palette, themes, and modal surfaces.
- Can complement the native UI tests once shell visuals stabilize.

Cons:
- High churn during active design iteration.
- Sensitive to OS rendering, fonts, scale factor, themes, and window size.
- Not ideal as the first GUI testing investment.

Best use:
- Later, for a small set of stable visual contracts.

### Option 5: Browser-Style/E2E Frameworks
Use tools like Playwright-style automation only for WebKit extension panes or generated plugin HTML surfaces.

Pros:
- Strong tooling if testing web content inside extension panes.
- Can be useful for plugin-rendered HTML in isolation.

Cons:
- OpenMUX is not browser-first; this would miss native AppKit and terminal interactions.
- Risks pushing architecture toward the wrong testing model.

Best use:
- Plugin/extension-pane HTML tests only, not general app GUI testing.

## Recommended Direction
Use a layered strategy:

1. Keep AppKit/controller tests as the primary GUI-adjacent test layer.
2. Add one small native `XCTest` UI suite for critical end-to-end GUI flows.
3. Extend smoke scripts only for launch/package/resource checks, not deep interaction coverage.
4. Defer screenshot testing until the shell visual language is more stable.
5. Use web testing only for plugin HTML surfaces, not the main app shell.

## First UI Test Scope
A first OpenMUX UI suite should be intentionally tiny:

- Launches `OpenMUXApp` with isolated `HOME` and `OMUX_HOME`.
- Disables state restoration with `-ApplePersistenceIgnoreState YES`.
- Uses a temporary config with deterministic theme and Ghostty options.
- Waits for the main window.
- Verifies the Workspace/Panes menu items exist and selected shortcuts are exposed.
- Opens command palette with `Cmd+Shift+P` and dismisses with Escape.
- Opens workspace search with `Cmd+P` and dismisses with Escape.
- Clicks stable accessibility-labeled controls such as create workspace or add pane tab, if those labels are reliable.

Avoid in the first pass:
- Pixel snapshots.
- Complex drag/drop.
- Terminal text assertions that depend on shell timing.
- Deep Ghostty behavior; that belongs behind `OmuxTerminalBridge` or upstream Ghostty tests.
- Large test matrices across keyboard layouts until the harness is stable.

## Architecture Fit
The UI tests should respect OpenMUX boundaries:

- Test OpenMUX-native shell behavior through menus, shortcuts, accessibility, and the public `omux`/control-plane surface.
- Do not reach into `libghostty` internals from UI tests.
- Keep terminal bridge validation mostly in `OmuxTerminalBridgeTests` and smoke launch tests.
- For keyboard correctness, include targeted UI cases only after the basic harness is reliable, especially Option/Alt, dead keys, IME, and EU/ISO layout-sensitive flows.

## Proposed Implementation Plan

1. Investigate the least invasive way to host an XCTest UI target for an SPM app.
   - Check whether Xcode can create a companion project without disrupting SPM as source of truth.
   - Alternative: create a separate `UITests` package/harness only if it can launch the built app reliably.

2. Define deterministic launch setup.
   - Build `OpenMUXApp` first.
   - Launch with isolated `HOME`, `OMUX_HOME`, and temporary config.
   - Disable restoration.
   - Set `GHOSTTY_RESOURCES_DIR` for local runs if needed.

3. Add stable accessibility identifiers/labels only where missing.
   - Prefer identifiers for test selectors where labels are user-facing/localized.
   - Keep accessibility useful for real users, not only tests.

4. Add 2-4 UI tests.
   - App launches and main window appears.
   - Command palette opens/closes by keyboard.
   - Workspace search opens/closes by keyboard.
   - Basic menu/shortcut exposure is present.

5. Add a Makefile entry only after the suite is reliable.
   - Example name: `make ui-test`.
   - Keep it separate from `make test` initially to avoid slowing normal verification.

6. Extend packaged smoke separately.
   - Keep `make smoke-packaged-release` focused on launch/resource regressions.
   - Optionally add one Accessibility-level window/menu existence check if CI permissions permit.

## Risks And Mitigations
- Flaky UI timing: use short explicit waits around stable UI elements; avoid arbitrary sleeps except in smoke scripts.
- CI GUI availability: keep UI tests opt-in at first; document local requirements before gating PRs.
- Accessibility permissions: prefer XCTest UI automation over raw System Events where possible.
- Localization: use accessibility identifiers for selectors; reserve labels for user-visible assertions.
- Terminal runtime timing: test shell chrome first; leave terminal fidelity to bridge tests and smoke tests.
- Packaged resource regressions: keep packaged smoke test in the release path regardless of UI test adoption.

## Decision Point
The main decision is whether to introduce an Xcode-hosted UI test target now or first expand script-level smoke testing. The recommended first choice is a native XCTest UI target if the project can tolerate a small companion Xcode test harness; otherwise, start with script-level packaged smoke probes and revisit XCTest UI when the project structure is ready.
