# Multi-Vendor Runtime State Detection for an OpenMUX AI Status Plugin

## Executive summary

OpenMUXÔÇÖs existing shape strongly favors **one external `ai-status` plugin host with multiple vendor-specific adapters**, rather than vendor logic inside core terminal code. The project already exposes a local CLI and JSON-RPC control plane, supports hooks and plugins, discovers executable user plugins from `~/.omux/plugins/`, and is explicitly designed to be ÔÇťterminal-first,ÔÇŁ ÔÇťinspectable,ÔÇŁ and open to automation. That makes a shared adapter host a natural fit: each adapter can observe vendor-native signals and emit a normalized pane state through `omux pane-status`, while OpenMUX core remains vendor-agnostic. Packaging-wise, this points toward **one installable `ai-status` plugin package**ÔÇölikely distributed through the OpenMUX plugin ecosystemÔÇöwith adapter-owned vendor modules inside it, not one plugin per vendor unless distribution or licensing forces a split. ţłÇciteţłéturn42view1ţłéturn42view4ţłü

The strongest current signals are not all the same kind. **Claude Code** has the richest officially documented lifecycle hooks and structured programmatic streaming; **Gemini CLI** has the clearest officially documented passive observer signals, because it explicitly documents dynamic terminal-title icons for Ready, Action Required, and Working, and it also offers hooks, streaming JSON, exit codes, and ACP JSON-RPC; **Codex CLI** has strong structured signals in `codex exec --json`, useful hooks, and a configurable terminal-title surface, but the *exact* title strings for interactive waiting states are less formally documented than GeminiÔÇÖs. ţłÇciteţłéturn19view2ţłéturn20view2ţłéturn20view3ţłéturn19view0ţłéturn30view0ţłéturn29view0ţłéturn29view2ţłéturn32search0ţłéturn7view3ţłéturn7view2ţłéturn7view0ţłü

For OpenMUX specifically, the best architecture is a **hybrid adapter model**. When OpenMUX launches or wraps a tool, adapters should prefer **native machine-readable streams** such as JSONL, JSON-RPC, or structured hooks. When OpenMUX is merely observing an already-running pane, adapters should fall back to **vendor-specific observer signals** such as terminal title changes, notification hooks, or documented local transcript/log files. The host should do the hard part once: debounce noisy inputs, dedupe repeated events, assign confidence, and map vendor events into the normalized states `working`, `needs-input`, `idle`, `error`, and a **host-synthesized** `clear`. Most vendors expose runtime or turn status; almost none expose a first-class `clear` state, so `clear` should be inferred by the host on session exit, explicit clear/reset actions, or loss of agent identity after a short grace period. This is also consistent with OpenMUXÔÇÖs external-plugin model and with the conceptual patterns visible in Mux0 and cmux. It also fits OpenMUX operationally: the app already coalesces noisy terminal-title churn, so a passive title observer can be practical as long as the adapter emits only meaningful state transitions instead of mirroring every raw title update. ţłÇciteţłéturn42view1ţłéturn41view0ţłéturn41view1ţłü

If you want a pragmatic v1, **Codex title changes are indeed one of the best zero-core-integration observer signals today**, especially because OpenMUX already coalesces title churn. But they are **not** the best overall signal if you control launch or can install a lightweight hook: ClaudeÔÇÖs lifecycle hooks and GeminiÔÇÖs documented stream/title/hook surfaces are stronger and more future-proof. The best first-wave targets are **Gemini CLI, Claude Code, and Codex CLI**, in that order for breadth of signal surface and documentation quality. At the same time, if OpenMUX ships only **one worked example first**, **Codex is still the most practical starter adapter** because its observer path can be validated directly against OpenMUXÔÇÖs existing `terminal.titleChanged` event stream and current shell behavior without requiring OpenMUX to own tool launch. GitHub Copilot CLI looks promising but needs one more round of source-level validation around prompt/permission events; Aider is workable mainly in wrapper mode; Cursor CLI is relevant, but the publicly gathered sources here do not yet show a stable status/event contract. ţłÇciteţłéturn25view2ţłéturn30view0ţłéturn32search0ţłéturn19view2ţłéturn20view2ţłéturn20view0ţłéturn7view3ţłéturn7view2ţłéturn39view0ţłéturn39view2ţłéturn35view2ţłéturn35view1ţłéturn41view2ţłü

## Detailed findings per vendor

**Codex CLI**

Codex has three distinct signal classes. First, there is a strong **machine-readable wrapper surface**: `codex exec --json` turns `stdout` into a JSONL event stream with documented event types including `thread.started`, `turn.started`, `turn.completed`, `turn.failed`, `item.*`, and `error`; the published example also shows an `item.started` event with `status: "in_progress"`. That is enough to map `working`, `idle`, and `error` very cleanly in wrapper mode. Second, there is an official **hook surface**: Codex documents events such as `PreToolUse`, `PermissionRequest`, `PostToolUse`, and `Stop`, which is enough to map `working`, `needs-input`, and `idle` even for interactive sessions if the user opts into a local hook handler. Third, there is a passive **presentation surface**: Codex officially supports `tui.terminal_title`, and documents that the terminal title items are configurable and can be disabled; a recent issue confirms that the title was being rewritten every ~100 ms while the spinner was active, which implies title churn is a normal part of the runtime signal. Codex also stores logs under `log_dir` and can persist history to `history.jsonl`; the Python SDK controls a local app-server over JSON-RPC, but that is only relevant if OpenMUX is the launcher, not a passive pane observer. ţłÇciteţłéturn7view3ţłéturn7view2ţłéturn7view0ţłéturn7view1ţłéturn7view4ţłéturn9search0ţłü

For OpenMUX, the safe conclusion is: **Codex is excellent in wrapper or hook mode, and good in title-observer mode**. The weak point is that the exact interactive title strings for waiting states are not presented as a stable public API in the docs gathered here. Your observed spinner frames and `Action Required` status are exactly the kind of signal an adapter can exploit, but they should be classified as **adapter-owned heuristics**, not a cross-vendor contract. Recent public issue activity around ÔÇťAction RequiredÔÇŁ titles reinforces that CodexÔÇÖs title behavior is evolving, even if it is useful. ţłÇciteţłéturn7view0ţłéturn9search0ţłéturn11search2ţłü

**Claude Code**

Claude has the strongest **interactive lifecycle instrumentation** in the set. In non-interactive mode, `claude -p --output-format stream-json --verbose --include-partial-messages` emits newline-delimited JSON events, and the docs explicitly show `stream_event` messages plus `system/api_retry` events. In the SDK, the agent loop exposes `SystemMessage`, `AssistantMessage`, `UserMessage`, `StreamEvent`, and `ResultMessage`, with `ResultMessage` marking end-of-loop and carrying a subtype that distinguishes success from limits or other stop conditions. That already gives a very strong wrapper surface. ţłÇciteţłéturn19view0ţłéturn19view1ţłéturn16view5ţłü

The more important OpenMUX signal surface, though, is ClaudeÔÇÖs **hook system**. Claude documents a rich event lifecycle with `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PermissionRequest`, `PermissionDenied`, `PostToolUse`, `PostToolUseFailure`, `Notification`, `Stop`, `StopFailure`, and more. The hook payloads are JSON and include fields such as `tool_name`, `tool_input`, `permission_mode`, `transcript_path`, `last_assistant_message`, and structured error fields. The docs show a `PermissionRequest` payload before a permission dialog is shown; they also show `Notification` payloads with `notification_type: "permission_prompt"`, message text, and title; they show `Stop` with the final assistant message; and they show `StopFailure` with error categories like `rate_limit` and `authentication_failed`. This is almost an ideal OpenMUX adapter surface because a user-installed hook can send high-confidence local state changes to `omux pane-status` without any screen scraping, keystroke capture, or title parsing. ţłÇciteţłéturn19view2ţłéturn20view2ţłéturn20view3ţłéturn20view1ţłéturn20view0ţłü

Claude also updates the terminal title, because Anthropic documents `CLAUDE_CODE_DISABLE_TERMINAL_TITLE=1` to disable automatic terminal-title updates based on conversation context. That proves title mutation exists, but in the gathered sources it does **not** prove a stable title-state contract comparable to GeminiÔÇÖs documented icons. So for Claude, **hooks and stream-json should be first-class**, while title parsing should remain optional and low-confidence. ţłÇciteţłéturn17view0ţłü

**Gemini CLI**

Gemini is the clearest case where a passive observer can still be robust. The configuration docs explicitly document `ui.dynamicWindowTitle` and say it updates the terminal title with current status icons: **Ready `ÔŚç`, Action Required `Ôťő`, Working `ÔťŽ`**. The same configuration surface documents `ui.showStatusInTitle`, which can include model thoughts during the working phase. Separately, GeminiÔÇÖs notifications docs explicitly say the CLI can send notifications for **Action required** when waiting for user input or tool approval, and **Session complete** when the session finishes successfully; those notifications are emitted via **OSC 9** where supported, with fallback to **BEL**. That is an unusually strong official observer surface for a terminal AI tool. ţłÇciteţłéturn29view0ţłéturn25view2ţłü

Gemini also has strong structured programmatic surfaces. Headless mode supports `--output-format json` and `--output-format stream-json`; the latter documents event types `init`, `message`, `tool_use`, `tool_result`, `error`, and `result`, and the headless docs document exit codes `0`, `1`, `42`, and `53`. GeminiÔÇÖs hook system includes a `Notification` event for system alerts such as tool permissions, and Gemini also offers **ACP mode**, a JSON-RPC protocol over stdio intended for IDE and developer-tool integrations. That means Gemini is good in wrapper mode, good in observer mode, and good in hybrid mode. ţłÇciteţłéturn30view0ţłéturn30view2ţłéturn32search0ţłü

The main caution is exactly the kind of caution OpenMUX should already expect: **title signals can be wrong at the edges**. A public Gemini issue reports that long-running or silent interactive shell commands can incorrectly show `Ôťő Action Required` in the terminal title when the correct state should be working. That does not make the title unusable; it means title parsing should be **debounced and cross-checked** with hook or lifecycle signals when available. ţłÇciteţłéturn31view1ţłü

**GitHub Copilot CLI**

GitHub Copilot CLI is now a GA terminal-native coding agent, and GitHub positions it as an autonomous tool with plan mode and autopilot mode. The docs also say Copilot CLI records every session locally, supports plugins, skills, hooks, and remote control, and can be steered from GitHub.com or GitHub Mobile. That is enough to say the platform is architecturally compatible with an OpenMUX adapter. ţłÇciteţłéturn39view0ţłéturn39view1ţłéturn39view4ţłü

The most concrete signal surface found here is the **hook surface**. GitHubÔÇÖs docs show repository-level and user-level hooks with examples for `sessionStart`, `userPromptSubmitted`, `agentStop`, and `sessionEnd`, stored under `~/.copilot/hooks/` or `$COPILOT_HOME/hooks/`. That makes completion and session-boundary detection feasible. However, in the sources gathered for this report, I did **not** find a documented prompt/permission hook event or a documented tool lifecycle stream equivalent to ClaudeÔÇÖs `PermissionRequest` or GeminiÔÇÖs `tool_use`/`tool_result`. The docs clearly indicate tool approvals exist, but the collected material here does not yet establish a robust machine-readable mapping for `needs-input` or a high-confidence `working` transition. So Copilot CLI belongs in **second-wave probing**, not the very first OpenMUX rollout. ţłÇciteţłéturn39view2ţłéturn39view1ţłéturn39view0ţłü

**Aider**

Aider is comparatively simple from a status-detection perspective. It offers useful **wrapper behavior**: `--message` or `--message-file` runs one instruction and exits; `--stream` controls textual output streaming; `--yes-always` auto-accepts confirmations; and `--notifications` or `--notifications-command` can tell you when the LLM has finished and is waiting for your input. Aider also has documented history files, including `.aider.chat.history.md`, `.aider.input.history`, and optional `.aider.llm.history`. ţłÇciteţłéturn35view2ţłéturn35view1ţłéturn36view0ţłéturn37view0ţłü

What Aider does **not** expose in the gathered sources is a formal JSONL event stream, terminal title protocol, hook lifecycle, JSON-RPC control plane, or other explicit runtime-state API aimed at external observers. So Aider can be integrated, but your adapter would mostly infer `working` from ÔÇťwrapper child is still running,ÔÇŁ `needs-input` from optional notifications or the return to the prompt, and `idle` from session readiness heuristics. That is functional, but it is exactly the class of brittle, tool-specific logic OpenMUX should keep at the adapter edge and avoid treating as authoritative. ţłÇciteţłéturn35view1ţłéturn35view2ţłéturn36view2ţłü

**Cursor CLI**

Cursor is now clearly relevant enough to monitor. Its public site advertises a terminal CLI, headless workflows, shell mode, command approval, and terminal integration for agents. However, the public sources gathered here did not expose a documented JSONL event taxonomy, hook contract, title protocol, or transcript/log schema suitable for normalized pane-state detection. That means Cursor should be considered **potentially promising but currently under-specified** for this specific OpenMUX problem. It is relevant, but not a first-wave adapter target without another pass through the docs. ţłÇciteţłéturn41view2ţłéturn40search11ţłéturn40search8ţłü

## External project analysis

OpenMUX itself already points toward the right shape. Its README describes a local CLI and JSON-RPC control plane, external hooks, plugin discovery from `~/.omux/plugins/`, extension panes, and official plugin and hook registries. That means the control plane you described is not just compatible with OpenMUXÔÇÖs architecture; it is almost perfectly aligned with it. A multi-vendor `ai-status` plugin can live entirely outside the app shell, remain inspectable and opt-in, and still feel native because it reports through the same public surfaces as everything else. ţłÇciteţłéturn42view1ţłéturn42view4ţłü

**Mux0** is useful mainly for its **state taxonomy**. Its public README says the app provides live AI agent status, specifically `running`, `idle`, `waiting-for-input`, and `finished`, and that each turn is tagged success or failed. That state model is conceptually close to the normalized status model you want for OpenMUX, and it shows that the UI value of such normalization is real. What the gathered public material does **not** reveal is how Mux0 obtains those statesÔÇöwhether from title parsing, hooks, process wrappers, logs, or some mixÔÇöso the reusable part here is the **behavioral model**, not the implementation. ţłÇciteţłéturn41view0ţłü

**cmux** is more informative architecturally. Its README explicitly says the notification system picks up **OSC 9/99/777** sequences and exposes a **`cmux notify`** CLI that can be wired into agent hooks for tools like Claude Code, OpenCode, and Codex. It also says the app has a socket API and CLI for control. That is exactly the design pattern OpenMUX should adopt: **a generic local control plane plus small vendor-neutral signal relays**, instead of burying vendor logic inside terminal rendering code. The tradeoff is that cmux, in the materials gathered here, appears more focused on **attention routing and notifications** than on exposing a full normalized finite-state machine with strong `working`/`idle`/`error` semantics. In other words, cmux proves the value of generic hook/OSC ingestion, while Mux0 proves the value of normalized AI-state presentation. OpenMUX can combine both ideas cleanly. ţłÇciteţłéturn41view1ţłéturn41view0ţłü

## Recommendation for OpenMUX

The best strategy is to build **one shared `ai-status` plugin host** with **adapter-owned vendor modules**. OpenMUX core should know only the normalized states and how to invoke `omux pane-status`; everything vendor-specific should live inside adapter manifests, matcher tables, and optional user-installed hook files. This matches OpenMUXÔÇÖs existing CLI/plugin/hook architecture and avoids private APIs, keyboard interception, or shell-bridge bloat. In practical terms, that means **one plugin package, many adapters**: a single install/discovery/config surface for users, with Codex/Gemini/Claude/Copilot modules inside it rather than a separate plugin per agent coder. ţłÇciteţłéturn42view1ţłéturn42view4ţłü

The ranked detection strategy should be:

1. **Native structured vendor signal** when OpenMUX controls launch or wrapper mode is acceptable. This includes Codex JSONL, Claude `stream-json` / SDK message streams, Gemini `stream-json`, and Gemini ACP JSON-RPC. These are the most robust and least ambiguous surfaces. ţłÇciteţłéturn7view3ţłéturn19view0ţłéturn19view1ţłéturn30view0ţłéturn32search0ţłü  
2. **Official local hooks** for interactive sessions. Claude and Codex are especially strong here; GeminiÔÇÖs hooks are also useful. Hooks preserve the terminal-first model and involve no keystroke interception. ţłÇciteţłéturn19view2ţłéturn20view2ţłéturn7view2ţłéturn30view2ţłü  
3. **Wrapper lifecycle plus exit/result semantics** for tools with weaker observer support. This is the best Aider path, and it is also useful for Cursor or Copilot if stronger event docs are not available yet. ţłÇciteţłéturn35view2ţłéturn30view0ţłü  
4. **Documented local logs, transcripts, or session files** as secondary observer inputs. CodexÔÇÖs log directory and history, ClaudeÔÇÖs transcript paths in hook payloads, AiderÔÇÖs history files, and CopilotÔÇÖs local session recording all fit here, though they are more privacy-sensitive and often less real-time. ţłÇciteţłéturn7view1ţłéturn19view3ţłéturn37view0ţłéturn39view0ţłü  
5. **Title and notification heuristics** for passive observation of pre-existing panes. This is a particularly good v1 path for Codex and Gemini because the signals are local, inspectable, cheap, and already compatible with OpenMUXÔÇÖs title-debounce behavior. GeminiÔÇÖs documented title icons make it the best example; Codex titles are still very useful but should be treated as less stable. ţłÇciteţłéturn29view0ţłéturn25view2ţłéturn7view0ţłéturn9search0ţłü  
6. **Bounded TUI/output heuristics** only as an opt-in last resort. They are the least portable and the most vulnerable to localization, theming, and UX churn. The Gemini ÔÇťfalse Action RequiredÔÇŁ issue is a good example of why UI-only interpretation should remain a fallback. ţłÇciteţłéturn31view1ţłü

The first-wave adapter targets should be **Gemini CLI**, **Claude Code**, and **Codex CLI**. Gemini gives you the best passive observer story today. Claude gives you the best hook-driven interactive story. Codex gives you a strong mixed mode that covers wrapper, hooks, logs, and titlesÔÇöand it remains the best **first concrete example** for the current OpenMUX change because it can be exercised against the existing `terminal.titleChanged` surface immediately. **GitHub Copilot CLI** should be the next target after one more pass through its hooks reference. **Aider** should be treated as wrapper-first and low-confidence observer mode. **Cursor CLI** should remain in the backlog until its status/event contracts are documented clearly enough to avoid speculative integration. ţłÇciteţłéturn29view0ţłéturn30view0ţłéturn19view2ţłéturn20view2ţłéturn7view3ţłéturn7view2ţłéturn39view2ţłéturn35view2ţłéturn41view2ţłü

A good first adapter contract is **event-oriented instead of state-oriented**. Each adapter should emit events such as `activity-started`, `approval-requested`, `approval-cleared`, `turn-succeeded`, `turn-failed`, `session-ended`, and `signal-lost`, plus metadata such as `vendor`, `pane_id`, `source_surface`, `confidence`, `timestamp`, and optional `detail`. The shared host should own the final state machine and the `clear` synthesis. That keeps vendor quirks from leaking into OpenMUX core and makes it easy to add new adapters later. This is an inference from the gathered signal surfaces and from OpenMUXÔÇÖs plugin/control-plane design. ţłÇciteţłéturn42view1ţłéturn42view4ţłéturn41view1ţłü

## Risks and unknowns

The biggest risk is **unstable private format drift**. Terminal title strings, undocumented JSON fields, and human-facing notifications can change without notice. Codex title behavior has already seen churn around animated spinner defaults, and Gemini has already had a title-state bug where `Action Required` appeared incorrectly during long-running shell work. Those are not reasons to avoid title signals; they are reasons to keep them adapter-owned, confidence-scored, and easy to override. ţłÇciteţłéturn9search0ţłéturn31view1ţłü

A second risk is **localization and presentation dependence**. Text such as ÔÇťPermission needed,ÔÇŁ ÔÇťAction required,ÔÇŁ or assistant summaries can change by locale, theme, or UX rewrite. This is why your design constraintÔÇöÔÇťdo not treat terminal title changes as authoritative command textÔÇŁÔÇöis correct. The safest observer signals are symbols, event types, explicit hook fields, and exit/result codes, not human-readable prose. GeminiÔÇÖs icon mapping is strong because it is symbolic and documented; ClaudeÔÇÖs notification prose is useful, but the typed hook fields are better. ţłÇciteţłéturn29view0ţłéturn20view3ţłü

A third risk is **privacy and security**. Reading logs, transcripts, or history files means reading prompts, code, shell output, and possibly secrets. Gemini explicitly documents environment-variable redaction for tool execution, but that does not remove the privacy sensitivity of transcript or log consumption. OpenMUX should keep log-reading opt-in, document paths clearly, and prefer hooks or structured runtime streams over transcript mining. ţłÇciteţłéturn30view1ţłéturn7view1ţłéturn37view0ţłü

A fourth risk is **incomplete observability for background subagents or remote flows**. ClaudeÔÇÖs public issue tracker has already raised an observability gap for background subagents in `stream-json`, and GitHub Copilot CLIÔÇÖs remote control model means some ÔÇťneeds inputÔÇŁ interactions may be fulfillable outside the local terminal, which can complicate local pane-state semantics. These are manageable, but they make a confidence model and a ÔÇťbest effortÔÇŁ label important. ţłÇciteţłéturn14search7ţłéturn39view1ţłü

The main unknowns after this research are straightforward: I did not gather the **full Copilot hooks reference event inventory**, I did not gather a **Cursor CLI event/schema reference**, and Mux0ÔÇÖs public materials did not reveal the **specific signal-acquisition implementation** behind its normalized status view. Those unknowns do not block a strong first wave, but they do block high-confidence claims for those areas. ţłÇciteţłéturn39view2ţłéturn41view2ţłéturn41view0ţłü

## Concrete next-step inputs for implementation

For **Codex**, the minimal stable pattern set to test first is small and high-value. In wrapper mode, test `codex exec --json` for `turn.started` or `item.started.status == "in_progress"` as `working`, `turn.completed` as `idle`, and `turn.failed` or `error` as `error`. In hook mode, test `PermissionRequest` as `needs-input`, `PreToolUse` as `working`, and `Stop` as `idle`. In pure observer mode, test only the title behaviors you already see in the wildÔÇöyour braille spinner frames for `working` and the observed `Action Required` title for `needs-input`ÔÇöbut classify those as heuristics because the docs gathered here only formally document title configurability, not exact title strings. Also respect the fact that title churn can be high. ţłÇciteţłéturn7view3ţłéturn7view2ţłéturn7view0ţłéturn9search0ţłü

For **Gemini**, probe the exact payloads for headless `stream-json` and ACP mode first, because those are likely to give you the cleanest adapter implementation. Then verify the title/icon behavior across plain terminals, tmux, and screen, and confirm how the `Notification` hook behaves for permission prompts and blocked execution. Gemini is promising because it has official support for title icons, hooks, notifications, structured JSON, exit codes, and ACPÔÇöso it may become the gold-standard adapter in both wrapper and observer modes. ţłÇciteţłéturn30view0ţłéturn29view0ţłéturn25view2ţłéturn30view2ţłéturn32search0ţłü

For **Claude**, probe the full hook coverage before doing anything else. The current evidence already supports `working`, `needs-input`, `idle`, and `error`, but the implementation should verify `PermissionRequest`, `Notification`, `Stop`, `StopFailure`, and any elicitation-related events in real sessions. Separately, confirm whether the interactive terminal title ever encodes a stable state contract or whether it is merely conversational context. Claude is likely strongest when the user opts into a tiny local hook installer. ţłÇciteţłéturn19view2ţłéturn20view2ţłéturn20view3ţłéturn20view1ţłéturn20view0ţłéturn17view0ţłü

For **GitHub Copilot CLI**, the next thing to probe is the official hooks reference itself: specifically whether there are hook events for tool execution, tool approval, agent-start, agent-progress, or prompt/approval waits. The gathered sources already show a useful session/hook/plugin architecture, but OpenMUX should not assume `needs-input` can be detected robustly until that reference is checked. ţłÇciteţłéturn39view0ţłéturn39view1ţłéturn39view2ţłü

The user-configurable pieces should be: which adapters are enabled; whether an adapter may install or suggest vendor hook files; title-regex overrides; stale-to-`clear` timeout; whether transcript/log reading is allowed; preferred mode (`observer`, `wrapper`, or `hybrid`) per vendor; and confidence thresholds for surfacing `error` versus falling back to `idle`.
