# Multi-Vendor Runtime State Detection for an OpenMUX AI Status Plugin

## Executive summary

OpenMUX’s existing shape strongly favors **one bundled Swift `ai-status` host with multiple vendor-specific adapters**, rather than vendor logic inside terminal rendering or the libghostty bridge. The project already exposes a local CLI and JSON-RPC control plane, supports hooks and plugins, and is explicitly designed to be “terminal-first,” “inspectable,” and open to automation. That makes a shared adapter host a natural fit: each adapter can observe vendor-native signals and emit a normalized pane state through `omux pane-status`, while OpenMUX core remains vendor-agnostic. Packaging-wise, this points toward a bundled first-party host that ships with OpenMUX, with adapter-owned vendor modules inside it. External plugin packages can still extend or replace the behavior later, but the initial implementation should not depend on a separate `omux-plugins` repository. citeturn42view1turn42view4

The strongest current signals are not all the same kind. **Claude Code** has the richest officially documented lifecycle hooks and structured programmatic streaming; **Gemini CLI** has the clearest officially documented passive observer signals, because it explicitly documents dynamic terminal-title icons for Ready, Action Required, and Working, and it also offers hooks, streaming JSON, exit codes, and ACP JSON-RPC; **Codex CLI** has strong structured signals in `codex exec --json`, useful hooks, and a configurable terminal-title surface, but the *exact* title strings for interactive waiting states are less formally documented than Gemini’s. citeturn19view2turn20view2turn20view3turn19view0turn30view0turn29view0turn29view2turn32search0turn7view3turn7view2turn7view0

For OpenMUX specifically, the best architecture is a **hybrid adapter model**. For normal interactive sessions, the next stronger phase after passive title detection should be **explicit vendor hook setup plus a local relay command**: vendor hooks call `omux ai-status hook --source <vendor> --event <event>` with vendor JSON on stdin, and the bundled host normalizes that input into pane status. When OpenMUX launches or wraps a tool, adapters should prefer **native machine-readable streams** such as JSONL, JSON-RPC, or structured hooks. When OpenMUX is merely observing an already-running pane and no hook is installed, adapters should fall back to **vendor-specific observer signals** such as terminal title changes, notification hooks, or documented local transcript/log files. The host should do the hard part once: debounce noisy inputs, dedupe repeated events, assign confidence, and map vendor events into the normalized states `working`, `needs-input`, `idle`, `error`, and a **host-synthesized** `clear`. Most vendors expose runtime or turn status; almost none expose a first-class `clear` state, so `clear` should be inferred by the host on session exit, explicit clear/reset actions, or loss of agent identity after a short grace period. This is also consistent with OpenMUX’s hook/control-plane model and with the concrete pattern visible in `manaflow-ai/cmux`. It also fits OpenMUX operationally: the app already coalesces noisy terminal-title churn, so a passive title observer can be practical as long as the adapter emits only meaningful state transitions instead of mirroring every raw title update. citeturn42view1turn41view0turn41view1

If you want a pragmatic v1, **Codex title changes are still one of the best zero-setup observer signals today**, especially because OpenMUX already coalesces title churn. But they are **not** the best overall signal if the user explicitly installs a lightweight hook: Claude’s lifecycle hooks and Gemini’s documented stream/title/hook surfaces are stronger and more future-proof, and cmux now confirms the viability of a hook relay model for Codex and Gemini in ordinary terminal sessions. The best first-wave targets are **Codex CLI, Gemini CLI, and Claude Code**. Codex remains the most practical starter adapter because its observer path can be validated directly against OpenMUX’s existing `terminal.titleChanged` event stream and current shell behavior, while the hook path gives a stronger follow-up for permissions and tool activity. Gemini has the best documented title/icon behavior and useful hooks. Claude has strong hook semantics, but OpenMUX should follow cmux’s conservative shape initially: use wrapper-injected or guided hook configuration rather than silently editing Claude-owned settings. JSONL wrappers are important for controlled launch, but they should be treated as secondary to hooks and passive observation for arbitrary interactive panes. GitHub Copilot CLI looks promising but needs one more round of source-level validation around prompt/permission events; Aider is workable mainly in wrapper mode; Cursor CLI is relevant, but the publicly gathered sources here do not yet show a stable status/event contract. citeturn25view2turn30view0turn32search0turn19view2turn20view2turn20view0turn7view3turn7view2turn39view0turn39view2turn35view2turn35view1turn41view2

## Detailed findings per vendor

**Codex CLI**

Codex has three distinct signal classes. First, there is a strong **machine-readable wrapper surface**: `codex exec --json` turns `stdout` into a JSONL event stream with documented event types including `thread.started`, `turn.started`, `turn.completed`, `turn.failed`, `item.*`, and `error`; the published example also shows an `item.started` event with `status: "in_progress"`. That is enough to map `working`, `idle`, and `error` very cleanly in wrapper mode. Second, there is an official **hook surface**: Codex documents events such as `PreToolUse`, `PermissionRequest`, `PostToolUse`, and `Stop`, which is enough to map `working`, `needs-input`, and `idle` even for interactive sessions if the user opts into a local hook handler. Third, there is a passive **presentation surface**: Codex officially supports `tui.terminal_title`, and documents that the terminal title items are configurable and can be disabled; a recent issue confirms that the title was being rewritten every ~100 ms while the spinner was active, which implies title churn is a normal part of the runtime signal. Codex also stores logs under `log_dir` and can persist history to `history.jsonl`; the Python SDK controls a local app-server over JSON-RPC, but that is only relevant if OpenMUX is the launcher, not a passive pane observer. citeturn7view3turn7view2turn7view0turn7view1turn7view4turn9search0

For OpenMUX, the safe conclusion is: **Codex is excellent in wrapper or hook mode, and good in title-observer mode**. The weak point is that the exact interactive title strings for waiting states are not presented as a stable public API in the docs gathered here. Your observed spinner frames and `Action Required` status are exactly the kind of signal an adapter can exploit, but they should be classified as **adapter-owned heuristics**, not a cross-vendor contract. Recent public issue activity around “Action Required” titles reinforces that Codex’s title behavior is evolving, even if it is useful. citeturn7view0turn9search0turn11search2

**Claude Code**

Claude has the strongest **interactive lifecycle instrumentation** in the set. In non-interactive mode, `claude -p --output-format stream-json --verbose --include-partial-messages` emits newline-delimited JSON events, and the docs explicitly show `stream_event` messages plus `system/api_retry` events. In the SDK, the agent loop exposes `SystemMessage`, `AssistantMessage`, `UserMessage`, `StreamEvent`, and `ResultMessage`, with `ResultMessage` marking end-of-loop and carrying a subtype that distinguishes success from limits or other stop conditions. That already gives a very strong wrapper surface. citeturn19view0turn19view1turn16view5

The more important OpenMUX signal surface, though, is Claude’s **hook system**. Claude documents a rich event lifecycle with `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PermissionRequest`, `PermissionDenied`, `PostToolUse`, `PostToolUseFailure`, `Notification`, `Stop`, `StopFailure`, and more. The hook payloads are JSON and include fields such as `tool_name`, `tool_input`, `permission_mode`, `transcript_path`, `last_assistant_message`, and structured error fields. The docs show a `PermissionRequest` payload before a permission dialog is shown; they also show `Notification` payloads with `notification_type: "permission_prompt"`, message text, and title; they show `Stop` with the final assistant message; and they show `StopFailure` with error categories like `rate_limit` and `authentication_failed`. This is almost an ideal OpenMUX adapter surface because a user-installed hook can send high-confidence local state changes to `omux pane-status` without any screen scraping, keystroke capture, or title parsing. citeturn19view2turn20view2turn20view3turn20view1turn20view0

Claude also updates the terminal title, because Anthropic documents `CLAUDE_CODE_DISABLE_TERMINAL_TITLE=1` to disable automatic terminal-title updates based on conversation context. That proves title mutation exists, but in the gathered sources it does **not** prove a stable title-state contract comparable to Gemini’s documented icons. So for Claude, **hooks and stream-json should be first-class**, while title parsing should remain optional and low-confidence. citeturn17view0

**Gemini CLI**

Gemini is the clearest case where a passive observer can still be robust. The configuration docs explicitly document `ui.dynamicWindowTitle` and say it updates the terminal title with current status icons: **Ready `◇`, Action Required `✋`, Working `✦`**. The same configuration surface documents `ui.showStatusInTitle`, which can include model thoughts during the working phase. Separately, Gemini’s notifications docs explicitly say the CLI can send notifications for **Action required** when waiting for user input or tool approval, and **Session complete** when the session finishes successfully; those notifications are emitted via **OSC 9** where supported, with fallback to **BEL**. That is an unusually strong official observer surface for a terminal AI tool. citeturn29view0turn25view2

Gemini also has strong structured programmatic surfaces. Headless mode supports `--output-format json` and `--output-format stream-json`; the latter documents event types `init`, `message`, `tool_use`, `tool_result`, `error`, and `result`, and the headless docs document exit codes `0`, `1`, `42`, and `53`. Gemini’s hook system includes a `Notification` event for system alerts such as tool permissions, and Gemini also offers **ACP mode**, a JSON-RPC protocol over stdio intended for IDE and developer-tool integrations. That means Gemini is good in wrapper mode, good in observer mode, and good in hybrid mode. citeturn30view0turn30view2turn32search0

The main caution is exactly the kind of caution OpenMUX should already expect: **title signals can be wrong at the edges**. A public Gemini issue reports that long-running or silent interactive shell commands can incorrectly show `✋ Action Required` in the terminal title when the correct state should be working. That does not make the title unusable; it means title parsing should be **debounced and cross-checked** with hook or lifecycle signals when available. citeturn31view1

**GitHub Copilot CLI**

GitHub Copilot CLI is now a GA terminal-native coding agent, and GitHub positions it as an autonomous tool with plan mode and autopilot mode. The docs also say Copilot CLI records every session locally, supports plugins, skills, hooks, and remote control, and can be steered from GitHub.com or GitHub Mobile. That is enough to say the platform is architecturally compatible with an OpenMUX adapter. citeturn39view0turn39view1turn39view4

The most concrete signal surface found here is the **hook surface**. GitHub’s docs show repository-level and user-level hooks with examples for `sessionStart`, `userPromptSubmitted`, `agentStop`, and `sessionEnd`, stored under `~/.copilot/hooks/` or `$COPILOT_HOME/hooks/`. That makes completion and session-boundary detection feasible. However, in the sources gathered for this report, I did **not** find a documented prompt/permission hook event or a documented tool lifecycle stream equivalent to Claude’s `PermissionRequest` or Gemini’s `tool_use`/`tool_result`. The docs clearly indicate tool approvals exist, but the collected material here does not yet establish a robust machine-readable mapping for `needs-input` or a high-confidence `working` transition. So Copilot CLI belongs in **second-wave probing**, not the very first OpenMUX rollout. citeturn39view2turn39view1turn39view0

**Aider**

Aider is comparatively simple from a status-detection perspective. It offers useful **wrapper behavior**: `--message` or `--message-file` runs one instruction and exits; `--stream` controls textual output streaming; `--yes-always` auto-accepts confirmations; and `--notifications` or `--notifications-command` can tell you when the LLM has finished and is waiting for your input. Aider also has documented history files, including `.aider.chat.history.md`, `.aider.input.history`, and optional `.aider.llm.history`. citeturn35view2turn35view1turn36view0turn37view0

What Aider does **not** expose in the gathered sources is a formal JSONL event stream, terminal title protocol, hook lifecycle, JSON-RPC control plane, or other explicit runtime-state API aimed at external observers. So Aider can be integrated, but your adapter would mostly infer `working` from “wrapper child is still running,” `needs-input` from optional notifications or the return to the prompt, and `idle` from session readiness heuristics. That is functional, but it is exactly the class of brittle, tool-specific logic OpenMUX should keep at the adapter edge and avoid treating as authoritative. citeturn35view1turn35view2turn36view2

**Cursor CLI**

Cursor is now clearly relevant enough to monitor. Its public site advertises a terminal CLI, headless workflows, shell mode, command approval, and terminal integration for agents. However, the public sources gathered here did not expose a documented JSONL event taxonomy, hook contract, title protocol, or transcript/log schema suitable for normalized pane-state detection. That means Cursor should be considered **potentially promising but currently under-specified** for this specific OpenMUX problem. It is relevant, but not a first-wave adapter target without another pass through the docs. citeturn41view2turn40search11turn40search8

## External project analysis

OpenMUX itself already points toward the right shape. Its README describes a local CLI and JSON-RPC control plane, external hooks, plugin discovery from `~/.omux/plugins/`, extension panes, and official plugin and hook registries. That means the control plane you described is not just compatible with OpenMUX’s architecture; it is almost perfectly aligned with it. A multi-vendor `ai-status` plugin can live entirely outside the app shell, remain inspectable and opt-in, and still feel native because it reports through the same public surfaces as everything else. citeturn42view1turn42view4

**Mux0** is useful mainly for its **state taxonomy**. Its public README says the app provides live AI agent status, specifically `running`, `idle`, `waiting-for-input`, and `finished`, and that each turn is tagged success or failed. That state model is conceptually close to the normalized status model you want for OpenMUX, and it shows that the UI value of such normalization is real. What the gathered public material does **not** reveal is how Mux0 obtains those states—whether from title parsing, hooks, process wrappers, logs, or some mix—so the reusable part here is the **behavioral model**, not the implementation. citeturn41view0

**manaflow-ai/cmux** is more informative architecturally than the initial reading suggested. The correct current finding is not that cmux performs hidden passive introspection of arbitrary agent TUIs. Its seamless behavior is built from **explicit hook setup, vendor config installation, and a socket relay**. The relevant source references are:

- `docs/feed.md`: documents `cmux hooks setup`, `cmux hooks setup --agent codex`, `cmux hooks uninstall`, the `cmux hooks feed --source <agent>` bridge, the `feed.push` V2 socket frame, the Feed sidebar, a JSONL audit log at `~/.cmuxterm/workstream.jsonl`, and 120 second soft-wait timeout behavior.
- `docs/agent-hooks.md`: documents supported agents and installed config files, including Codex `~/.codex/hooks.json` and `~/.codex/config.toml`, Gemini `~/.gemini/settings.json`, and Claude Code through wrapper-injected settings.
- `CLI/CMUXCLI+AgentHookDefinitions.swift`: defines the generic hook integration model, per-agent config metadata, feed-hook events, nested/flat config formats, Codex config TOML post-install behavior, and comments that feed hooks invoke `cmux hooks feed --source <agent>` with a roughly 120 second timeout.
- `docs/notifications.md`: documents attention routing through OSC 9/99/777 and `cmux notify`, which complements Feed but is not the same thing as a normalized runtime-state API.

The concrete pattern OpenMUX should adopt from cmux is: **a user-run setup command writes only explicit OpenMUX-owned hook entries into vendor configuration; vendor hooks pipe JSON into a small CLI relay; the relay forwards events to a local socket/control-plane host; the host records/audits events and maps actionable states for the UI; timeouts fall back to the vendor’s own prompt rather than freezing the workflow**. cmux’s Feed is mainly an approval/decision and activity surface, not a full normalized finite-state machine with stable `working`/`idle`/`error` semantics. Mux0 remains useful for the state taxonomy; cmux is the stronger reference for hook installation, relay mechanics, and timeout behavior. citeturn41view1turn41view0

## Recommendation for OpenMUX

The best strategy is to build **one bundled Swift `ai-status` host** with **adapter-owned vendor modules**. OpenMUX core should know only the normalized states and how to invoke/report pane status; everything vendor-specific should live inside adapter matchers, hook normalizers, config installers, and optional wrapper parsers. This matches OpenMUX’s existing CLI/hook/control-plane architecture and avoids private APIs, keyboard interception, or shell-bridge bloat. In practical terms, that means **one first-party host, many adapters**: a single command and config surface for users, with Codex/Gemini/Claude modules inside it rather than separate plugins per agent coder. External plugins can still build on the same public reporting surface later. citeturn42view1turn42view4

The ranked detection strategy should be:

1. **Official local hooks for interactive sessions** after explicit setup. Claude, Codex, and Gemini all have useful hook surfaces, and cmux confirms this is practical for ordinary terminal panes when a CLI relay normalizes stdin and posts to a local host. Hooks preserve the terminal-first model and involve no keystroke interception. citeturn19view2turn20view2turn7view2turn30view2
2. **Title and notification heuristics** for zero-setup passive observation of pre-existing panes. This is a particularly good v1 fallback for Codex and Gemini because the signals are local, inspectable, cheap, and already compatible with OpenMUX’s title-debounce behavior. Gemini’s documented title icons make it the best example; Codex titles are still very useful but should be treated as less stable. citeturn29view0turn25view2turn7view0turn9search0
3. **Native structured vendor signal when OpenMUX controls launch or wrapper mode is acceptable.** This includes Codex JSONL, Claude `stream-json` / SDK message streams, Gemini `stream-json`, and Gemini ACP JSON-RPC. These are robust and unambiguous, but they only help the sessions OpenMUX launches through a wrapper. citeturn7view3turn19view0turn19view1turn30view0turn32search0
4. **Wrapper lifecycle plus exit/result semantics** for controlled-launch sessions or tools with weaker observer support. This is the best Aider path, and it is also useful for Cursor or Copilot if stronger event docs are not available yet. citeturn35view2turn30view0
5. **Documented local logs, transcripts, or session files** as secondary observer inputs. Codex’s log directory and history, Claude’s transcript paths in hook payloads, Aider’s history files, and Copilot’s local session recording all fit here, though they are more privacy-sensitive and often less real-time. citeturn7view1turn19view3turn37view0turn39view0
6. **Bounded TUI/output heuristics** only as an opt-in last resort. They are the least portable and the most vulnerable to localization, theming, and UX churn. The Gemini “false Action Required” issue is a good example of why UI-only interpretation should remain a fallback. citeturn31view1

The first-wave adapter targets should be **Gemini CLI**, **Claude Code**, and **Codex CLI**. Gemini gives you the best passive observer story today. Claude gives you the best hook-driven interactive story. Codex gives you a strong mixed mode that covers wrapper, hooks, logs, and titles—and it remains the best **first concrete example** for the current OpenMUX change because it can be exercised against the existing `terminal.titleChanged` surface immediately. **GitHub Copilot CLI** should be the next target after one more pass through its hooks reference. **Aider** should be treated as wrapper-first and low-confidence observer mode. **Cursor CLI** should remain in the backlog until its status/event contracts are documented clearly enough to avoid speculative integration. citeturn29view0turn30view0turn19view2turn20view2turn7view3turn7view2turn39view2turn35view2turn41view2

A good first adapter contract is **event-oriented instead of state-oriented**. Each adapter should emit events such as `activity-started`, `approval-requested`, `approval-cleared`, `turn-succeeded`, `turn-failed`, `session-ended`, and `signal-lost`, plus metadata such as `vendor`, `pane_id`, `source_surface`, `confidence`, `timestamp`, and optional `detail`. The shared host should own the final state machine and the `clear` synthesis. The first public CLI shape should include `omux ai-status hooks setup|uninstall [codex|claude|gemini]` for explicit vendor config management and `omux ai-status hook --source <vendor> --event <event>` as the hook relay that reads vendor JSON from stdin. That keeps vendor quirks from leaking into OpenMUX core and makes it easy to add new adapters later. This is an inference from the gathered signal surfaces and from OpenMUX’s plugin/control-plane design. citeturn42view1turn42view4turn41view1

## Risks and unknowns

The biggest risk is **unstable private format drift**. Terminal title strings, undocumented JSON fields, and human-facing notifications can change without notice. Codex title behavior has already seen churn around animated spinner defaults, and Gemini has already had a title-state bug where `Action Required` appeared incorrectly during long-running shell work. Those are not reasons to avoid title signals; they are reasons to keep them adapter-owned, confidence-scored, and easy to override. citeturn9search0turn31view1

A second risk is **localization and presentation dependence**. Text such as “Permission needed,” “Action required,” or assistant summaries can change by locale, theme, or UX rewrite. This is why your design constraint—“do not treat terminal title changes as authoritative command text”—is correct. The safest observer signals are symbols, event types, explicit hook fields, and exit/result codes, not human-readable prose. Gemini’s icon mapping is strong because it is symbolic and documented; Claude’s notification prose is useful, but the typed hook fields are better. citeturn29view0turn20view3

A third risk is **privacy and security**. Reading logs, transcripts, or history files means reading prompts, code, shell output, and possibly secrets. Gemini explicitly documents environment-variable redaction for tool execution, but that does not remove the privacy sensitivity of transcript or log consumption. OpenMUX should keep log-reading opt-in, document paths clearly, and prefer hooks or structured runtime streams over transcript mining. citeturn30view1turn7view1turn37view0

A fourth risk is **incomplete observability for background subagents or remote flows**. Claude’s public issue tracker has already raised an observability gap for background subagents in `stream-json`, and GitHub Copilot CLI’s remote control model means some “needs input” interactions may be fulfillable outside the local terminal, which can complicate local pane-state semantics. These are manageable, but they make a confidence model and a “best effort” label important. citeturn14search7turn39view1

The main unknowns after this research are straightforward: I did not gather the **full Copilot hooks reference event inventory**, I did not gather a **Cursor CLI event/schema reference**, and Mux0’s public materials did not reveal the **specific signal-acquisition implementation** behind its normalized status view. Those unknowns do not block a strong first wave, but they do block high-confidence claims for those areas. citeturn39view2turn41view2turn41view0

## Concrete next-step inputs for implementation

For **Codex**, the minimal stable pattern set to test first is small and high-value. In wrapper mode, test `codex exec --json` for `turn.started` or `item.started.status == "in_progress"` as `working`, `turn.completed` as `idle`, and `turn.failed` or `error` as `error`. In hook mode, test `PermissionRequest` as `needs-input`, `PreToolUse` as `working`, and `Stop` as `idle`. In pure observer mode, test only the title behaviors you already see in the wild—your braille spinner frames for `working` and the observed `Action Required` title for `needs-input`—but classify those as heuristics because the docs gathered here only formally document title configurability, not exact title strings. Also respect the fact that title churn can be high. citeturn7view3turn7view2turn7view0turn9search0

For **Gemini**, probe the exact payloads for headless `stream-json` and ACP mode first, because those are likely to give you the cleanest adapter implementation. Then verify the title/icon behavior across plain terminals, tmux, and screen, and confirm how the `Notification` hook behaves for permission prompts and blocked execution. Gemini is promising because it has official support for title icons, hooks, notifications, structured JSON, exit codes, and ACP—so it may become the gold-standard adapter in both wrapper and observer modes. citeturn30view0turn29view0turn25view2turn30view2turn32search0
